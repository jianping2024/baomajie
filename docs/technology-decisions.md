# 宝马街技术栈职责、选型理由与后端决策

本文档解释每项技术负责什么、为什么使用，并记录后端正式采用NestJS组合方案的决策。Laravel内容作为备选方案和决策背景保留。

## 0. 已确认的架构决策

决定日期：2026年6月23日

状态：已接受

正式后端组合：

```text
NestJS + Fastify Adapter
PostgreSQL + Prisma
Redis + BullMQ
Meilisearch
独立Next.js管理后台
S3/R2 + Stripe
```

决定理由：

- 前后端统一使用TypeScript，降低团队跨语言维护成本。
- NestJS模块、Provider、Guard、Pipe和Interceptor适合建立明确工程边界。
- 消息、通知和未来实时能力更容易在Node.js体系内扩展。
- Prisma提供类型安全的数据访问和可审查的数据库迁移。
- 管理后台采用独立Next.js应用，获得更强的定制能力，并复用UI和OpenAPI客户端。
- 旧宝马街仅作为数据迁移来源，不要求复用旧PHP业务代码。
- 第一版只通过公开HTML抓取内容，不迁移旧用户、密码或私有状态。

接受的代价：

- 需要自行建设Cookie Session、CSRF、Token轮换和权限框架。
- 需要自行开发管理后台，不能依赖Filament快速生成。
- BullMQ监控、通知抽象和文件处理流水线需要明确实施。
- PostGIS复杂查询可能使用参数化原生SQL，不能完全依赖Prisma抽象。
- 公开页面抓取无法恢复真实用户归属和完整历史记录。

除非出现重大约束变化，例如必须长期复用旧PHP代码，否则开发过程中不再重新选择后端框架。

## 1. 选型原则

技术选择服务于以下产品目标：

1. 公开资讯和分类信息必须有良好SEO。
2. 移动网页需要具备接近App的使用体验。
3. 后续能够复用主要前端代码发布iOS和Android。
4. 平台长期包含账号、审核、服务商认证、消息和支付；第一版只实现基础消息，不实现支付。
5. 首发葡萄牙，但数据和路由支持逐国扩展。
6. MVP团队规模有限，避免不必要的分布式系统复杂度。
7. 现有宝马街是PHP网站，旧数据和旧逻辑可能需要渐进迁移。

不以“技术是否流行”作为首要标准，而以交付速度、正确性、招聘维护成本和产品适配度作为标准。

## 3. Next.js

### 负责的功能

- 公开首页。
- 资讯、指南、分类信息和服务商页面。
- 服务端渲染和静态生成。
- SEO元数据、Sitemap和结构化数据。
- 登录后的个人中心、发布、消息和工作台界面。
- PWA Manifest和Service Worker集成。
- 按国家和语言生成路由。
- 调用后端API并显示数据。

### 采用理由

宝马街同时需要“可被搜索引擎收录的网站”和“像App一样操作的登录产品”。Next.js可以在一个前端项目里同时处理：

- 公开页面的SSR/ISR。
- 登录后页面的动态交互。
- React组件复用。
- 路由、缓存和SEO。
- PWA与未来Capacitor封装。

使用纯React SPA会弱化首屏和SEO；使用传统服务端模板则会增加Web App复杂交互的开发成本。

### 主要代价

- Server Components、缓存和失效机制需要团队理解。
- 如果滥用Client Components，包体和性能会恶化。
- Next.js不能替代稳定的业务后端和数据库事务层。

## 4. React与TypeScript

### 负责的功能

- 列表、表单、导航、消息等交互组件。
- 前端状态和用户操作。
- 编译期类型检查。
- 根据OpenAPI生成API请求和响应类型。
- Web和Capacitor App复用UI及业务交互。

### 采用理由

- Next.js的标准UI运行时。
- 生态和工程人才充足。
- 适合复杂表单、发布流程和服务商工作台。
- TypeScript可以减少接口字段变化造成的错误。

### 需要说明

TypeScript只提供编译期安全。所有外部输入仍需在后端验证，前端也需要Zod等运行时校验。

## 5. Tailwind CSS与无样式组件

### 负责的功能

- 移动端和桌面端响应式布局。
- 颜色、间距、字体和状态样式。
- 统一按钮、表单、卡片、弹窗和导航。
- 深色模式和主题变量。

### 采用理由

- 适合从设计规范快速建立共享组件。
- 减少大型全局CSS文件和命名冲突。
- 容易保持网站和PWA界面一致。
- 与Next.js生态配合成熟。

Radix UI或React Aria负责弹窗、菜单、焦点和键盘行为，Tailwind只负责外观。

### 主要代价

- 页面直接堆积大量class会降低可读性。
- 必须通过共享组件约束，不能让每个页面自行发明样式。

## 6. 后端应用框架

后端负责：

- 用户、身份和权限。
- 分类信息状态和审核。
- 企业及服务商认证。
- 需求、匹配、报价和预约。
- 消息、通知和举报。
- 支付订单和订阅。
- 数据库事务。
- 搜索索引同步。
- 文件处理。
- 队列和定时任务。
- 旧系统数据迁移。
- 管理后台。

本项目正式采用NestJS。Laravel同样能够完成系统，以下分析保留用于说明为何在当前约束下选择NestJS。

## 7. 为什么原方案采用Laravel

### 7.1 与旧宝马街技术接近

现有网站响应头显示为PHP系统。即使新系统不复用旧代码，PHP团队通常更容易：

- 阅读旧逻辑。
- 理解旧密码Hash和会话。
- 编写旧数据库迁移工具。
- 在迁移期增加只读导出接口。
- 处理旧PHP系统和新API的并行运行。

这降低的是迁移风险，不是因为PHP在语言层面优于TypeScript。

### 7.2 业务后台交付速度

宝马街需要大量运营后台：

- 分类和地区管理。
- 资讯编辑。
- 分类信息审核。
- 服务商资质审核。
- 举报处理。
- 用户封禁。
- 订单查看。
- 数据迁移监控。

Laravel配合Filament可以较快完成这些标准后台功能。NestJS本身不提供同等级的一体化后台方案，通常还需要：

- 单独开发React Admin/Next.js后台。
- 或选择AdminJS等工具并处理适配。

如果团队本来就计划定制整个后台，Laravel的这项优势会减弱。

### 7.3 账号、队列、定时任务和支付生态

Laravel内置或官方生态覆盖：

- Sanctum认证。
- Policy和Gate权限。
- Queue与Horizon。
- Scheduler。
- Notifications。
- Mail。
- Filesystem/S3。
- Cashier/Stripe。
- Eloquent和数据库迁移。

这些恰好是宝马街MVP的大部分后端需求，默认约定较完整。

### 7.4 CRUD和审核型业务适配

宝马街的大部分核心负载是：

- 表单提交。
- 状态流转。
- 内容审核。
- 列表查询。
- 数据导入。
- 支付和通知。

这类系统的主要难点是业务正确性和运营效率，而不是极高并发的长连接。Laravel足以处理，并且能较快形成稳定的模块化单体。

### 7.5 招聘和维护

如果现有维护人员熟悉PHP，Laravel能降低交接成本。如果未来团队以TypeScript工程师为主，则这一结论可能反转。

## 8. 为什么可以采用NestJS

NestJS不是不适合。以下条件下，它可能是更合理的选择。

### 8.1 团队以TypeScript为核心

如果前后端开发人员都熟悉TypeScript、Node.js和NestJS：

- 前后端共享语言和工具。
- DTO、枚举和部分验证模型更容易保持一致。
- 招聘范围可以统一。
- 开发人员可以跨前后端协作。

但共享语言不等于可以共享所有业务代码。数据库模型、API DTO和前端表单模型仍应保持边界。

### 8.2 实时功能是核心产品

如果产品明确把以下功能作为早期重点：

- 高频即时聊天。
- 在线状态。
- 实时竞价。
- 大量WebSocket连接。
- 实时协作。

NestJS在Node事件模型、WebSocket Gateway和实时服务组织方面更自然。

当前规划中的消息只是服务撮合辅助能力，SSE或普通WebSocket均可实现，因此实时能力还不足以单独决定选择NestJS。

### 8.3 需要统一工程规范

NestJS明确提供：

- Module
- Controller
- Provider
- Guard
- Interceptor
- Pipe
- Decorator

对于熟悉Angular式依赖注入和模块结构的团队，它能形成严格、统一的工程组织。

### 8.4 后期拆分服务

如果平台未来确实需要拆分：

- 消息服务。
- 通知服务。
- 搜索同步服务。
- 支付服务。

NestJS拥有较成熟的消息传输和微服务抽象。不过MVP不应因为“将来也许需要微服务”提前增加复杂度。

## 9. Laravel与NestJS对比

| 维度 | Laravel | NestJS |
|---|---|---|
| 主要语言 | PHP | TypeScript/Node.js |
| 与旧宝马街PHP迁移 | 更直接 | 需要跨语言迁移 |
| 标准CRUD开发速度 | 很快 | 快 |
| 审核运营后台 | Filament优势明显 | 通常另建后台 |
| 认证、队列、通知 | 官方约定完整 | 需要组合多个包 |
| 实时/WebSocket | 可以实现 | 更自然 |
| 前后端统一语言 | 否 | 是 |
| 类型系统 | PHP类型+静态分析 | TypeScript体验更统一 |
| 数据库ORM | Eloquent | Prisma/TypeORM/MikroORM需选择 |
| 长任务 | Queue Worker | BullMQ Worker |
| 结构约束 | 灵活，需团队规范 | 框架约束更强 |
| 冷启动/内存 | PHP-FPM请求模型成熟 | Node常驻进程 |
| 迁移脚本 | 很适合旧PHP/MySQL数据 | 同样可做，但无旧栈优势 |
| 招聘适配 | 取决于当地PHP人才 | TS全栈人才通常较广 |
| MVP总体复杂度 | 较低 | 中等 |
| 自定义复杂后台 | 可以 | TS全栈可能更顺手 |

性能不是这个项目选择两者的主要依据。经过正确缓存、索引和队列设计后，两者都足够支撑MVP及相当规模的业务。

## 10. 如果改用NestJS，具体技术组合

建议不要只把Laravel名称替换为NestJS，还要确定完整配套。

### 10.1 推荐组合

| 能力 | NestJS方案 |
|---|---|
| HTTP | NestJS + Fastify Adapter |
| ORM | Prisma |
| 数据库 | PostgreSQL |
| 数据校验 | Zod，或class-validator/class-transformer |
| API文档 | `@nestjs/swagger` |
| 认证 | Passport策略+自建会话/Token服务 |
| Cookie Session | Redis Session Store |
| 队列 | BullMQ |
| 定时任务 | `@nestjs/schedule` |
| 实时 | SSE或Nest WebSocket Gateway |
| 文件 | AWS SDK S3 Client |
| 支付 | Stripe Node SDK |
| 搜索 | Meilisearch JS Client |
| 邮件 | 邮件供应商SDK或Nodemailer |
| 后台 | Next.js Admin或React Admin |
| 权限 | Guards + Policies/CASL |
| 日志 | Pino |
| 测试 | Vitest/Jest + Supertest |

### 10.2 目录结构

```text
apps/api/src/
├── modules/
│   ├── identity/
│   ├── geography/
│   ├── content/
│   ├── listings/
│   ├── providers/
│   ├── marketplace/
│   ├── messaging/
│   ├── trust/
│   ├── billing/
│   ├── media/
│   ├── search/
│   └── migration/
├── common/
│   ├── auth/
│   ├── database/
│   ├── errors/
│   ├── observability/
│   └── validation/
├── app.module.ts
└── main.ts
```

### 10.3 Prisma的作用

Prisma负责：

- 数据库schema和migration。
- 类型安全的数据库查询。
- 数据库客户端。

Prisma不负责：

- 业务状态机。
- API DTO。
- 权限。
- 搜索索引。
- 分布式事务。

复杂报表和PostGIS查询可能需要原生SQL。必须在选型验证阶段制作地理和全文查询原型。

### 10.4 NestJS需要额外建设的能力

相比Laravel方案，需要明确实现：

- Cookie会话和CSRF策略。
- Refresh token轮换。
- 管理后台。
- 通知渠道抽象。
- 文件处理流水线。
- Policy模型。
- Horizon类似的队列运营界面。
- 统一异常格式和事务边界。

这些都能实现，但不能低估工程量。

## 11. 后端决策结论

### 选择Laravel，如果：

- 能取得或需要维护现有PHP代码。
- 团队有可靠的PHP/Laravel能力。
- 希望尽快交付运营后台。
- MVP重点是内容、审核、服务撮合和支付。
- 团队较小，希望采用约定完整的一体化框架。

### 选择NestJS，如果：

- 新团队主要是TypeScript/NestJS工程师。
- 不准备复用旧PHP代码，只迁移数据库。
- 愿意单独建设管理后台。
- 即时消息和实时交互是产品早期核心。
- 团队能长期维护认证、队列、权限等组合式基础设施。

### 本项目结论

采用NestJS组合方案，前提是：

- 新系统不复用旧PHP业务代码。
- 旧站通过数据库、导出API或公开页面适配器迁移数据。
- 管理后台按产品需求独立开发。
- 团队接受自行建设认证、权限、队列监控和通知基础设施。

Laravel保留为历史备选，不再作为当前实施候选。

最终选择应在阶段0增加一次后端技术验证，而不是立即编码。

## 12. NestJS技术基线验证任务

不再进行Laravel与NestJS竞赛式验证。编码第一阶段使用NestJS制作一个小型垂直切片，验证组合方案的关键风险：

必须实现：

1. 用户Cookie登录。
2. 创建分类信息。
3. 数据库事务和状态流转。
4. 图片直传凭证。
5. 队列更新Meilisearch。
6. 管理员审核记录。
7. OpenAPI文档。
8. 一个旧数据导入命令。

验收指标：

- 实际开发时间。
- 代码复杂度。
- 自动测试难度。
- 后台完成度。
- 队列和失败任务可观测性。
- 团队代码评审结果。
- 部署和运行资源。
- 两名非作者工程师能够理解和修改代码。
- Prisma能够满足常规事务，PostGIS查询有明确原生SQL方案。
- BullMQ失败任务可查看、重试并告警。

验证失败时应调整局部组件，不因单项问题重新推翻NestJS后端决策。

## 13. PostgreSQL

### 负责的功能

- 用户、内容、分类信息、服务商和订单等权威数据。
- 事务和数据一致性。
- 唯一约束和外键。
- 审计记录。
- 地区和距离查询。
- JSONB动态属性。

### 采用理由

- 关系模型适合平台业务。
- 事务能力适合报价接受、订单和支付状态。
- PostGIS支持地区和距离。
- JSONB可容纳分类动态字段，但仍能保留核心关系结构。
- 全文搜索索引损坏时，可从PostgreSQL恢复权威数据。

### 为什么不是MongoDB

宝马街的主要数据关系明确，并且有大量事务、唯一约束、审核和报表需求。MongoDB不会减少这些关系，反而会把一致性处理移到应用层。

## 14. Redis

### 负责的功能

- Web会话。
- API限流。
- 短期缓存。
- 验证码。
- 队列。
- 消息在线事件。
- 幂等键短期结果。

### 采用理由

这些数据访问频繁、寿命短或需要原子计数，不适合全部压在PostgreSQL上。

### 使用边界

Redis不是权威业务数据库。Redis数据丢失不能导致订单、报价或消息正文永久丢失。

## 15. Meilisearch

### 负责的功能

- 中文关键词搜索。
- 拼写容错。
- 分类、地区、价格和时间筛选。
- 资讯、分类信息和服务商搜索排序。

### 采用理由

平台搜索不仅是SQL模糊匹配，还需要多字段检索、筛选和较快响应。Meilisearch比自行维护PostgreSQL搜索逻辑更容易起步，运维复杂度低于Elasticsearch/OpenSearch。

### 使用边界

- 搜索结果必须能够从PostgreSQL重建。
- 私密字段不得进入索引。
- 索引同步采用outbox和队列，不在HTTP请求中强耦合。

## 16. S3或Cloudflare R2

### 负责的功能

- 分类信息图片。
- 文章封面。
- 消息附件。
- 服务商资质文件。
- 缩略图和处理后图片。

### 采用理由

- 应用服务器不需要保存本地文件。
- 易于CDN分发和扩容。
- 可以使用预签名URL直传。
- 公开图片和私密证件可以分离权限。

R2出口成本可能更低；AWS S3生态和合规选项更成熟。最终根据部署云和欧盟数据区域决定。

## 17. Meilisearch以外为什么不直接用Elasticsearch

Elasticsearch/OpenSearch适合更复杂的搜索、日志和大型集群，但MVP会增加：

- 集群运维。
- 内存成本。
- Mapping和升级复杂度。
- 搜索调优门槛。

如果后续出现复杂同义词、多语言分词、大规模日志分析或千万级索引需求，再评估迁移。

## 18. Stripe

### 负责的功能

- 服务商会员付款。
- 订阅续费。
- 线索和置顶订单付款。
- 退款。
- 支付方式和银行卡安全处理。

### 采用理由

- 欧洲支付生态成熟。
- 支持订阅、Webhook和税务相关扩展。
- 宝马街无需存储银行卡数据。

### 使用边界

Stripe只决定支付是否成功；宝马街数据库决定用户购买了什么、获得了什么权限以及退款业务规则。

## 19. Cloudflare

### 负责的功能

- DNS和TLS。
- 静态资源CDN。
- WAF和DDoS基础防护。
- 页面及图片缓存。
- Bot和速率限制辅助。
- 管理后台访问保护。

### 采用理由

现有网站已经使用Cloudflare，继续使用可以减少DNS和安全迁移风险。

Cloudflare不能替代应用层权限、CSRF、输入校验和审计。

## 20. Capacitor

### 负责的功能

- 把Web应用装入iOS和Android容器。
- 调用相机、相册、推送、定位和安全存储。
- 处理App深度链接。

### 采用理由

第一阶段先验证PWA，产品稳定后复用前端代码上架App，避免同时开发Web、iOS和Android三套产品。

### 使用边界

如果未来出现大量原生动画、后台定位或重度设备能力，Capacitor可能不够，需要独立React Native或原生客户端。当前宝马街业务不需要提前承担该成本。

## 21. Docker Compose

### 负责的功能

- 启动本地PostgreSQL、Redis、Meilisearch、MinIO和邮件测试服务。
- 固定团队开发依赖版本。
- 简化新开发人员环境准备。

### 采用理由

避免每名开发者手工安装不同版本的基础服务。

生产可以使用容器，但Docker Compose本身不作为生产编排方案。

## 22. OpenAPI

### 负责的功能

- 定义请求、响应和错误格式。
- 生成前端TypeScript客户端。
- 提供接口文档。
- 检测破坏性接口变化。

### 采用理由

Next.js和独立后端之间需要明确契约。无论后端使用Laravel还是NestJS，OpenAPI都能减少手写类型漂移。

## 23. Sentry、日志和指标

### 负责的功能

- 记录前后端异常。
- 关联request ID。
- 发现慢接口和失败队列。
- 支付和迁移失败告警。

### 采用理由

平台涉及异步队列、搜索同步、图片处理和支付，仅靠服务器文本日志难以定位跨系统问题。

监控数据需要控制个人信息，不能把消息正文、证件和支付敏感数据上传到第三方错误平台。

## 24. 最终技术组合

前端、数据库、搜索、文件和基础设施选择可以保持不变：

```text
Next.js + React + TypeScript
PostgreSQL + Redis + Meilisearch
S3/R2 + Cloudflare
Stripe + Capacitor
```

后端已确定为：

```text
NestJS + Fastify Adapter
Prisma + PostgreSQL
Redis + BullMQ
独立Next.js管理后台
```

Laravel对比内容仅用于保存选型依据。项目实施、估算、招聘和代码规范均以NestJS方案为准。
