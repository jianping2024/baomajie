# 宝马街第一版开发实施步骤

## 1. 第一版目标

第一版主要完成：

> 把现有宝马街公开页面的数据抓取并写入新系统，使用Next.js和NestJS制作一个新的响应式首页、分类列表、基础详情页和基础站内消息，为后续服务商撮合、支付和App封装建立技术基础。

第一版不是完整重做所有旧功能，也不是完整服务交易平台。

## 2. 第一版最终交付内容

### 2.1 公开网站

- 葡萄牙新版首页。
- 手机和桌面响应式布局。
- 分类导航。
- 最新信息流。
- 推荐或置顶信息区域。
- 分类信息列表。
- 分类信息基础详情页。
- 搜索入口和基础搜索结果。
- 公开联系方式展示。
- 原始数据来源说明。
- 举报和申请下架入口。
- SEO标题、描述、canonical和Sitemap。

### 2.2 数据抓取

- 抓取现有宝马街公开首页。
- 抓取选定分类列表。
- 抓取公开详情页。
- 解析标题、分类、地区、正文、时间、图片和公开联系方式。
- 数据清洗。
- 重复内容识别。
- 图片下载和去重。
- 全量抓取。
- 增量更新。
- 失败记录和人工检查后台。

### 2.3 新系统后端

- NestJS API。
- Prisma数据层。
- PostgreSQL数据库。
- Redis缓存和BullMQ抓取队列。
- Meilisearch全文搜索。
- S3或R2图片存储。
- OpenAPI接口文档。
- 抓取记录和来源映射。
- 最小管理后台API。

### 2.4 用户和消息

- 新用户注册。
- 新用户登录和退出。
- 基础个人资料。
- 一对一文本消息。
- 会话列表。
- 未读数量。
- 拉黑和举报。

抓取且未被认领的旧信息不能接收站内消息，只展示原公开联系方式。

### 2.5 PWA基础

- Web App Manifest。
- App图标。
- 添加到手机主屏幕。
- 独立显示模式。
- 基础离线页面。

第一版不包括App Store或Google Play上架。

## 3. 第一版明确不开发

- 在线支付。
- Stripe接入。
- 会员套餐。
- 线索购买。
- 付费置顶。
- 服务商报价。
- 服务预约。
- 完整服务商认证。
- 旧用户和密码迁移。
- 旧收藏和历史消息迁移。
- 原生iOS和Android应用。
- 多国同时上线。

## 4. 实施阶段概览

第一版分为九个步骤：

| 步骤 | 阶段 | 主要结果 |
|---|---|---|
| 1 | 页面和数据审计 | 确认能抓什么以及如何映射 |
| 2 | 首页原型和设计系统 | 确认新版首页长什么样 |
| 3 | 基础工程 | 建立可运行、可测试、可部署的项目 |
| 4 | 数据库和抓取器 | 把旧站公开数据稳定导入新系统 |
| 5 | 新版首页 | 用新数据库数据展示首页 |
| 6 | 列表、详情和搜索 | 建立完整公开浏览路径 |
| 7 | 注册和基础消息 | 支持新用户沟通 |
| 8 | 管理、SEO、PWA和安全 | 达到上线标准 |
| 9 | 全量抓取和灰度上线 | 正式验证第一版 |

## 5. 步骤一：页面和数据审计

预计时间：3至5个工作日

### 目标

在编码抓取器前，确认旧站公开页面的结构、字段和变化情况。

### 工作内容

1. 枚举首页、分类页和详情页URL。
2. 统计不同详情页模板。
3. 检查分页和筛选参数。
4. 确认每类页面的DOM定位方式。
5. 确认可抓取字段：
   - 标题
   - 描述
   - 分类
   - 地区
   - 价格或薪资
   - 发布时间
   - 图片
   - 电话
   - 微信
   - WhatsApp
   - 来源URL
6. 检查登录限制、验证码和访问频率限制。
7. 抽取100至500条页面进行准确率测试。
8. 识别重复信息和更新信息的判断方式。
9. 确认哪些分类进入第一版。
10. 建立旧分类和地区到新系统的映射表。

### 交付物

- 页面模板清单。
- 抓取字段字典。
- 分类和地区映射表。
- 测试抓取样本。
- 解析准确率报告。
- 抓取频率和边界说明。

### 验收标准

- 首页、目标分类和详情页均有稳定解析方案。
- 每条样本都能追溯到原始URL。
- 同一页面重复抓取不会生成两条数据。
- 明确记录无法可靠抓取的字段。

## 6. 步骤二：首页原型和设计系统

预计时间：4至7个工作日

### 目标

在开发首页前确认信息架构、视觉层级和移动端交互，让首页成为“生活问题入口”而不是信息墙。

### 主要视觉对标

首页以[Expatica Portugal](https://www.expatica.com/pt/)为主要信息架构和视觉参考，但不复制其品牌或页面。信息组织只借鉴其克制、清晰、分层的方式，语义要改成在葡萄牙生活的中国人的问题入口。

重点借鉴：

- 国家门户式首屏和清晰栏目导航。
- 指南、推荐内容和实用工具分区。
- 大图头条与卡片信息流组合。
- 桌面多栏、移动端单栏。
- Directory、Jobs和Housing等服务入口与内容结合。

宝马街需要保留的差异：

- 首页首先回答“来葡萄牙后先解决什么问题”。
- 高优先级入口是居留、税号、开户、找房、找工作和找服务。
- 发布、服务、广告和订阅是首页的商业导向，不是平铺的功能噪音。
- 推荐、普通信息和商业内容必须明确区分。
- 增加站内消息、公开联系方式和防诈骗提示。
- 面向葡萄牙场景下的中国用户采用中文优先的信息表达。

### 首页建议模块

1. 问题搜索和发布入口。
2. 重要公告或防诈骗提醒。
3. 核心问题入口。
4. 今日解决方案。
5. 最新可联系内容。
6. 订阅和工具入口。
7. 服务商和广告位。
8. 葡萄牙资讯或实用指南入口。
9. 安装到手机提示。

### 设计任务

- 手机首页。
- 桌面首页。
- 信息卡片。
- 推荐和普通信息的视觉区分。
- 分类图标。
- 图片缺失状态。
- 加载骨架。
- 空状态和错误状态。
- 深色模式是否首发。
- 底部导航。

### 交付物

- 首页高保真原型。
- 手机和桌面稿。
- 颜色、字体和间距规范。
- 按钮、卡片、标签、表单组件。
- 页面交互说明。
- 可本地运行的首页原型页面，便于先看视觉和信息架构。

### 验收标准

- 首页在360、390、768、1024和1440像素布局明确。
- 推荐内容、普通内容和商业内容容易区分。
- 用户在首页两次点击以内能进入主要分类详情。
- 原型版可以直接本地启动并打开查看。

## 7. 步骤三：基础工程

预计时间：3至5个工作日

### 目标

建立统一、可重复运行的开发环境。

### 工程内容

```text

apps/api       NestJS API和抓取任务

packages/ui    共享组件
packages/api-client
```

基础服务：

- PostgreSQL。
- Redis。
- Meilisearch。
- MinIO或S3/R2开发环境。
- Mailpit。

### 开发任务

1. 建立pnpm workspace。
2. 配置TypeScript strict。
3. 配置ESLint和格式化。
4. 建立NestJS Fastify项目。
5. 建立Prisma。
6. 建立Next.js App Router项目。
7. 建立Docker Compose。
8. 建立环境变量规范。
9. 建立OpenAPI生成流程。
10. 建立CI。
11. 接入结构化日志和Sentry。

### 交付物

- `make setup`或等价初始化命令。
- 本地开发说明。
- 可运行的Web、API和后台空项目。
- 自动化检查和构建。

### 验收标准

- 新电脑按照README可以启动全部服务。
- Pull Request自动执行类型检查、Lint、测试和生产构建。
- API健康检查和数据库连接正常。

## 8. 步骤四：数据库和公开页面抓取器

预计时间：7至12个工作日

### 目标

稳定地把旧站公开数据转换成新系统的标准记录。

### 首版核心数据表

- countries
- regions
- categories
- listings
- listing_media
- media
- import_jobs
- import_job_errors
- legacy_mappings
- crawl_pages
- audit_logs

用户和消息表可以在此阶段创建，但功能在步骤七完成。

### 抓取架构

```text
BullMQ抓取任务
  → 下载公开HTML
  → 站点页面类型识别
  → Cheerio解析
  → Zod验证中间DTO
  → HTML白名单清洗
  → 分类和地区映射
  → 内容校验值
  → Prisma事务写入
  → 图片异步下载
  → 写入Meilisearch Outbox
```

### 技术任务

1. 建立`PublicHtmlSource`适配器。
2. 建立首页、分类页和详情页解析器。
3. 建立统一中间DTO。
4. 建立分类和地区映射。
5. 建立URL及旧ID去重。
6. 建立内容校验值更新判断。
7. 建立HTML清洗规则。
8. 建立图片下载、哈希和去重。
9. 建立限速、重试和失败队列。
10. 建立全量抓取命令。
11. 建立增量抓取命令。
12. 建立页面消失后的失效规则。
13. 建立抓取统计接口。

### 推荐抓取频率

- 首页：每5至15分钟。
- 分类第一页：每15至30分钟。
- 历史分页：每日或每周。
- 已导入详情：根据更新时间或分批复查。

实际频率需要根据旧站承载能力调整。

### 交付物

- 可重复执行的抓取CLI。
- BullMQ抓取Worker。
- 抓取统计。
- 错误列表。
- 数据清洗结果。
- Meilisearch索引同步。

### 验收标准

- 抓取中断后可以继续。
- 重复执行不产生重复信息。
- 失败页面可以重试。
- 每条记录保留来源URL和抓取时间。
- 抓取任务不会阻塞网站API。
- 达到审计阶段约定的解析准确率。

## 9. 步骤五：新版首页

预计时间：5至8个工作日

### 目标

从新PostgreSQL和NestJS API读取数据，完成第一版核心页面。

### API

建议首页聚合接口：

```text
GET /v1/home
```

返回：

- 分类。
- 推荐信息。
- 最新信息。
- 招聘信息。
- 房产信息。
- 商业服务。
- 公告。

首页也可以拆分接口，但不能为每张卡片单独发请求。

### 前端任务

1. 首页Header。
2. 搜索入口。
3. 分类快捷入口。
4. 推荐信息。
5. 最新信息流。
6. 分类区块。
7. 加载和错误状态。
8. 桌面多栏布局。
9. 移动端底部导航。
10. 图片优化。
11. 分享。
12. 安装PWA提示预留。

### 后端任务

1. 首页聚合查询。
2. 公开信息过滤。
3. 推荐排序规则。
4. 缓存和失效。
5. 接口限流。
6. OpenAPI文档。

### 当前实现状态

- 已完成 NestJS `GET /v1/home` 聚合接口。
- 已完成 Next.js 首页改造，首页改为读取 API 数据并提供本地兜底数据。
- 当前首页仍然使用种子数据，后续会替换为 Prisma 查询与真实导入数据。

### 验收标准

- 首页完全从新系统读取数据。
- 旧站不可访问时，已导入内容仍能展示。
- 首页接口P95低于500毫秒。
- 首屏LCP目标低于2.5秒。
- 手机和桌面端通过视觉验收。

### 当前实现状态

- 已完成 `GET /v1/listings` 列表接口。
- 已完成 `GET /v1/listings/:slug` 详情接口。
- 已完成 `GET /v1/search` 搜索接口。
- 已完成 `/pt/listings` 列表页。
- 已完成 `/pt/listings/[slug]` 详情页。
- 已完成 `/pt/search` 搜索页。
- 当前数据仍为种子数据，后续切换为 Prisma 查询和导入数据。

- 按地区筛选。

- title和description。
- canonical。
- Breadcrumb结构化数据。
- Sitemap。
- robots规则。

### 验收标准

## 11. 步骤七：新用户和基础站内消息

预计时间：5至8个工作日

### 目标

支持新系统用户之间的基础沟通，但不为未认领旧信息制造虚假接收人。

### 账号功能

- 邮箱注册。
- 邮箱验证。
- 登录和退出。
- 找回密码。
- Cookie Session。
- CSRF。
- 基础个人资料。

手机验证码可根据短信成本后置。

### 消息功能

- 用户从新系统原生发布的信息发起会话。
- 一对一文本消息。
- 会话列表。
- 未读数量。
- SSE或轮询获取新消息。
- 拉黑。
- 举报。
- 基础邮件提醒。

### 特殊规则

```text
原生发布且有owner_user_id
  → 可以发起站内消息

抓取导入且尚未认领
  → 禁止发起站内消息
  → 只显示原公开联系方式
```

### 验收标准

- 用户A和用户B可以完成一对一文本沟通。
- 未登录用户会被引导登录。
- 被拉黑后不能继续发送。
- 未认领旧信息不会出现无接收人的消息按钮。

- 新用户注册和基础消息可用。

## 14. 推荐排期

部分工作可以并行，建议总体为6至10周。

| 周期 | 工作 |
|---|---|
| 第1周 | 页面审计、字段映射、首页原型 |
| 第2周 | 设计系统、基础工程、数据库 |
| 第3周 | 抓取器和数据清洗 |
| 第4周 | 图片、搜索同步、新版首页 |
| 第5周 | 首页完善、列表和详情 |
| 第6周 | 搜索、SEO、账号和消息 |
| 第7周 | 管理后台、PWA和安全 |
| 第8周 | 全量抓取、测试和灰度上线 |
| 第9至10周 | 风险缓冲和质量修复 |

如果只有一名全职开发人员，应按10至14周估算。

## 15. 第一版关键依赖

开发开始前需要确认：

1. 首页抓取哪些公开分类。
2. 抓取多少历史分页。
3. 是否抓取公开图片到新存储。
4. 哪些联系方式可以重新展示。
5. 首页视觉风格。
6. 新用户使用邮箱还是手机号作为主要登录方式。
7. 新系统是否允许第一版发布新信息。

第7项会明显影响第一版范围。如果允许新发布，还需要开发：

- 发布表单。
- 图片上传。
- 内容审核。
- 短信或邮箱激活。
- 信息有效期和下架。

## 16. 完成定义

第一版只有在以下条件全部满足时才算完成：

- 公开抓取数据稳定进入新数据库。
- 新首页不依赖旧站实时响应。
- 首页、列表、详情和搜索完整。
- 手机和桌面端通过验收。
- 新用户登录和基础消息可用。
- 抓取旧信息的用户归属规则正确。
- 联系方式展示和反滥用规则生效。
- 管理员可以下架错误或被举报的信息。
- SEO、性能、安全和监控达到约定标准。
- 部署和恢复流程有文档并经过验证。
