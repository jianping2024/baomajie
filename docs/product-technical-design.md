# 宝马街产品与技术设计

## 1. 产品目标

宝马街不再只是分类广告列表，而是面向在葡萄牙生活的中国人的“生活问题入口”。平台先用真实问题承接搜索流量，再把用户导向资讯、工具、服务、广告和订阅，形成可持续的内容与商业闭环。

首阶段目标：

- 在葡萄牙验证产品和商业模式。
- 保留现有分类信息业务及其搜索流量。
- 增加资讯、办事指南、服务商认证和服务需求撮合。
- 使用同一套业务、数据和前端能力支持网站、PWA 和未来 App。
- 技术和数据结构支持后续扩展到西班牙、法国、德国等国家。

## 2. 产品定位

一句话定位：

> 帮助在葡萄牙生活的中国人快速解决落地、办事、找房、找工作和找服务这些现实问题。

核心业务链路：

```text
搜索引擎、微信、社交媒体
            ↓
生活问题入口
            ↓
资讯 / 指南 / 清单 / 工具
            ↓
服务页 / 广告位 / 订阅入口
            ↓
咨询、联系、下单或订阅
            ↓
复访、转化和复购
```

## 3. MVP产品范围

第一版采用分阶段MVP。MVP-1先完成公开数据抓取、问题入口型首页、列表和基础详情；MVP-2再完善服务商撮合、广告和订阅。具体已确认范围见[第一版范围决策](./mvp-decisions.md)。

### 3.1 公开网站

- 首页
- 生活问题入口
- 资讯与办事指南
- 分类信息列表与详情
- 服务商列表与详情
- 地区及分类聚合页
- 搜索和筛选
- 登录注册
- 发布入口

### 3.2 用户端

- 个人资料
- 发布、编辑和下架分类信息
- 基础站内消息
- 收藏
- 评价与举报
- 通知

### 3.3 服务商端

- 个人或企业服务商主页
- 服务类别及服务地区
- 第一版仅建立数据模型和基础展示
- 完整资质、匹配、报价和预约后置

### 3.4 管理后台

- 用户管理
- 企业与服务商认证
- 分类信息审核
- 资讯和指南编辑
- 分类和地区管理
- 举报与评价审核
- 订单、套餐和推广管理
- 广告位和订阅位管理
- 导入任务与迁移状态
- 审计日志

### 3.5 MVP暂缓功能

- 在线支付、订单和订阅
- 服务需求报价与预约闭环
- 完整服务商认证
- 旧用户和密码迁移
- 社区论坛
- 视频和直播
- 平台担保交易
- 完整即时通讯
- 多国同时上线
- 多币种结算
- AI法律或医疗结论
- 第一阶段直接上架原生 App

### 3.6 第一版数据来源

- 不接入旧数据库或只读导出API。
- 不迁移旧用户、密码、收藏和消息。
- 只抓取旧站无需登录即可访问的公开页面。
- 抓取数据保留原始URL、旧记录标识、抓取时间和内容校验值。
- 抓取内容统一归属于系统导入主体，发布者完成认领前不绑定新账号。
- 新首页从新数据库读取数据，不在用户访问首页时实时抓取旧站。

## 4. 信息架构

首页视觉与内容组织以Expatica Portugal为主要参考，但产品语义要改成“生活问题入口”。首页要先回答用户在葡萄牙最常遇到的问题，再把流量导向解决方案、服务、广告和订阅。宝马街不复制其品牌，重点增加中文问题导向、公开联系方式和防诈骗机制。

推荐保留主域名和国家路径：

```text
www.baomajie.com/pt/
www.baomajie.com/pt/news/
www.baomajie.com/pt/guides/
www.baomajie.com/pt/listings/
www.baomajie.com/pt/services/
www.baomajie.com/pt/requests/
www.baomajie.com/pt/app/

api.baomajie.com/
admin.baomajie.com/
```

第一阶段不建议把 Web App 完全拆到独立子域名，以减少跨域登录、重复页面和部署复杂度。产品成熟后可以再评估 `app.baomajie.com`。

## 5. 移动端导航

登录后的移动端采用五个底部入口：

1. 首页
2. 资讯
3. 发布
4. 消息
5. 我的

“发布”根据用户身份提供：

- 发布分类信息
- 提交服务需求
- 发布服务报价

## 6. Web、PWA和App路线

### 第一阶段：响应式网站和PWA

- 公开页面支持搜索引擎收录。
- 登录后页面采用接近原生 App 的交互。
- 支持添加到手机桌面。
- 支持全屏启动、基础离线页、草稿保存和 Web Push。
- 图片在客户端压缩后上传。
- 支持微信、WhatsApp 和系统分享。

### 第二阶段：原生封装

业务稳定后使用 Capacitor 封装：

- iOS App
- Android App
- APNs/Firebase 原生推送
- 相机、相册和定位
- 深度链接
- 应用商店支付规则适配

原则是复用业务逻辑和大部分界面，不维护三套独立产品。

## 7. 推荐技术架构

```text
Next.js Web/PWA
       │
       │ HTTPS REST API
       ▼
NestJS API与业务服务
       │
       ├── PostgreSQL
       ├── Redis
       ├── Meilisearch
       ├── S3兼容对象存储
       ├── 邮件/短信/推送
       └── Stripe

Next.js 独立管理后台
```

## 8. 技术选型

| 模块 | 推荐技术 | 选择原因 |
|---|---|---|
| 前端 | Next.js、TypeScript | SSR、SEO、响应式网站和PWA统一 |
| UI | Tailwind CSS、Headless组件 | 适合建立统一设计系统 |
| 后端 | NestJS、Fastify、Prisma | 全TypeScript工程栈、模块化结构、实时能力和长期团队统一 |
| 数据库 | PostgreSQL | 地理、多语言、JSON和复杂查询能力较强 |
| 缓存与队列 | Redis | 缓存、限流、验证码、异步任务 |
| 搜索 | Meilisearch | 中文全文搜索和筛选易于实施 |
| 后台 | 独立Next.js后台 | 与公开前端共享组件和类型，同时保持权限及部署隔离 |
| 文件 | S3兼容对象存储 | 图片和私密文件统一管理 |
| CDN与安全 | Cloudflare | CDN、WAF、缓存和基础防护 |
| 支付 | Stripe | 欧洲支付、订阅和发票能力成熟 |
| App封装 | Capacitor | 复用Web产品 |
| API规范 | OpenAPI | 前后端契约和接口测试 |
| 监控 | Sentry、结构化日志 | 前后端错误追踪 |

## 9. 模块边界

后端可先采用模块化单体，不建议第一阶段使用微服务。

建议模块：

- Identity：账号、登录和权限
- Geography：国家、地区和语言
- Content：资讯和指南
- Listings：分类信息
- Providers：企业、服务商和资质
- Marketplace：需求、匹配、报价和预约
- Messaging：会话和消息
- Trust：认证、评价、举报和审核
- Billing：套餐、订单和支付
- Media：图片、文件和访问控制
- Search：索引和搜索
- Migration：旧数据导入和映射
- Admin：运营后台和审计

模块之间通过应用服务调用，避免直接跨模块修改数据。

## 10. API设计原则

第一阶段使用版本化REST API：

```text
GET    /v1/articles
GET    /v1/listings
GET    /v1/listings/{id}
POST   /v1/listings
PATCH  /v1/listings/{id}

GET    /v1/providers
GET    /v1/providers/{id}
POST   /v1/service-requests
POST   /v1/service-requests/{id}/quotes

GET    /v1/conversations
POST   /v1/conversations/{id}/messages

POST   /v1/media
POST   /v1/reports
```

统一要求：

- API版本号
- 游标或页码分页
- 请求参数校验
- RBAC和资源所有权检查
- 速率限制
- 写操作幂等键
- 错误代码规范
- 审计日志
- OpenAPI文档

## 11. 权限设计

使用RBAC加资源所有权：

| 角色 | 主要权限 |
|---|---|
| 游客 | 浏览公开内容 |
| 普通用户 | 发布、提交需求、收藏、沟通和举报 |
| 服务商 | 管理企业资料、查看需求和报价 |
| 编辑 | 创建和维护资讯及指南 |
| 审核员 | 审核信息、资质、评价和举报 |
| 管理员 | 用户、支付、权限及系统配置 |

敏感后台操作必须记录操作人、时间、目标记录和变更内容。

## 12. SEO设计

- 公开内容使用SSR或静态生成。
- 每条资讯、指南、分类信息和服务商都有独立URL。
- 提供canonical和hreflang。
- 使用Article、Breadcrumb、LocalBusiness等结构化数据。
- Sitemap按国家和内容类型拆分。
- 列表筛选参数设置规范化URL，避免重复收录。
- 删除页面按情况返回404、410或301。
- 登录、消息和个人中心禁止搜索引擎收录。
- 迁移期间建立旧URL到新URL的永久重定向表。

## 13. 安全和GDPR

- 收集数据前说明用途和合法依据。
- 支持Cookie同意管理。
- 支持用户数据导出、修正和删除。
- 管理后台强制双因素认证。
- 身份证件和资质文件使用私有存储及签名URL。
- 密码使用Argon2id或bcrypt。
- 敏感字段加密。
- 上传文件执行类型检查、病毒扫描和EXIF清理。
- 日志中不得记录密码、验证码和完整证件内容。
- 对注册、登录、发布、消息和支付接口限流。
- 建立封禁、拉黑、举报和内容审核机制。

## 14. 性能目标

- 首页LCP目标低于2.5秒。
- 常规API查询P95目标低于500毫秒。
- 图片输出WebP或AVIF。
- 列表强制分页。
- 搜索请求使用专用搜索索引。
- 数据库慢查询纳入监控。
- 静态资源由CDN提供。
- 避免首页直接输出过多列表和脚本。

## 15. 商业化设计

初期收入来源：

1. 服务商月度或年度会员。
2. 服务商购买有效客户线索。
3. 分类信息置顶和推广。
4. 企业品牌主页。
5. 标准化服务的合规佣金。

法律、金融、保险和房地产相关收费必须按国家进行合规审查。

## 16. 关键产品指标

- 注册转化率
- 分类信息发布完成率
- 搜索无结果率
- 服务需求提交率
- 服务商报价率
- 首次响应时间
- 用户查看报价率
- 成交或预约率
- 服务商续费率
- 举报率和审核时长
- 迁移用户登录成功率
