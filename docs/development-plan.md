# 宝马街开发、迁移与上线计划

## 1. 计划目标

以葡萄牙为首发市场。第一版交付公开数据抓取、新版响应式首页、分类列表、基础详情、基础站内消息和最小管理后台。在线支付、完整服务撮合和旧用户迁移后置。

第一版建议周期为6至10周，取决于旧站页面类型、解析复杂度和首页设计范围。完整服务撮合平台仍按后续阶段单独估算。

## 2. 项目阶段

## 阶段0：产品决策与需求确认

预计时间：1周

### 工作步骤

1. 明确平台定位和首发国家。
2. 确定MVP功能和明确不做事项。
3. 定义游客、用户、服务商、编辑、审核员和管理员角色。
4. 绘制注册、发布、需求提交、报价、预约、举报流程。
5. 确定首发分类和服务类别。
6. 联系方式按公开规则实现，并提供用户控制、举报和反滥用措施。
7. 服务商允许个人和企业；完整认证后置。
8. 第一版不含支付，包含基础站内消息。
9. 第一版不迁移旧用户和密码，只导入公开内容。
10. 编写验收标准。

### 交付物

- PRD
- MVP范围清单
- 用户流程图
- 角色权限矩阵
- 内容审核规则
- 商业化规则草案

### 退出条件

- 所有关键产品决策有明确负责人和结论。
- MVP没有相互冲突的业务规则。

## 阶段1：公开页面抓取审计

预计时间：1至2周

### 所需输入

- 现有公开首页和分类页面。
- 公开详情页样本。
- 当前分类、地区和URL规则。
- 允许抓取和展示的公开字段范围。

### 工作步骤

1. 枚举首页、分类列表和详情页模板。
2. 检查分页、筛选和旧URL规则。
3. 确认标题、正文、分类、地区、时间、图片和联系方式的DOM来源。
4. 检查结构变化和缺失字段。
5. 建立公开页面字段到新数据模型的映射。
6. 选取100至500条代表性页面试抓取。
7. 测试内容校验值和重复识别。
8. 测试图片下载、校验和去重。
9. 形成清洗、合并、失效和下架规则。
10. 输出抓取风险和解析准确率报告。

### 交付物

- 页面模板审计报告
- 抓取字段映射表
- 数据质量报告
- 测试抓取结果
- URL重定向策略

### 退出条件

- 每个目标公开字段都有抓取、转换或放弃规则。
- 测试抓取可重复执行且不会产生重复记录。

## 阶段2：交互原型与设计系统

预计时间：2周

### 工作步骤

1. 建立站点地图。
2. 设计移动端底部导航。
3. 设计桌面端和移动端公开首页。
4. 设计资讯列表、详情和指南页面。
5. 设计分类信息列表、详情和发布流程。
6. 设计服务商列表、详情和认证流程。
7. 设计服务需求问卷和报价流程。
8. 设计消息、收藏、通知和个人中心。
9. 设计服务商工作台。
10. 设计管理后台关键审核页面。
11. 建立颜色、字体、间距和组件规范。
12. 补充加载、空状态、错误和权限不足状态。
13. 使用交互原型走通核心流程。

### 交付物

- 高保真交互原型
- 响应式页面规范
- 设计组件库
- 核心页面验收清单

### 退出条件

- 用户和服务商核心任务可在原型中完成。
- 移动端和桌面端布局经过评审。

## 阶段3：基础工程

预计时间：1周

### 工作步骤

1. 建立代码仓库和分支策略。
2. 建立Next.js前端项目。
3. 建立NestJS API项目和Prisma数据层。
4. 建立PostgreSQL数据库迁移。
5. 配置Redis。
6. 配置对象存储和CDN。
7. 配置本地、测试、预发布和生产环境。
8. 建立OpenAPI规范。
9. 建立CI检查、测试和构建流程。
10. 配置错误监控和结构化日志。
11. 配置密钥和环境变量管理。
12. 建立数据库备份与恢复流程。

### 交付物

- 可部署的基础项目
- 自动化构建
- 环境说明
- API规范骨架
- 监控和日志入口

## 阶段4：账号、权限和媒体

预计时间：1至2周

### 工作步骤

1. 邮箱或手机注册。
2. 登录、退出和刷新令牌。
3. 找回密码。
4. 邮箱和手机号验证。
5. 用户资料和头像。
6. RBAC和资源所有权。
7. 管理员双因素认证。
8. 账号封禁和恢复。
9. 图片上传、压缩和缩略图。
10. 私密文件签名访问。
11. GDPR数据导出和账号删除流程。
12. 写操作审计日志。

### 测试重点

- 越权访问
- 暴力登录
- 重复账号
- 文件类型伪造
- 私密文件泄露

## 阶段5：资讯与分类信息

预计时间：2至3周

### 工作步骤

1. 国家、地区、分类和动态属性管理。
2. 资讯和指南编辑发布。
3. 文章版本记录。
4. 分类信息创建、编辑和草稿。
5. 图片排序。
6. 审核、拒绝和重新提交。
7. 自动过期和下架。
8. 搜索、筛选和排序。
9. 收藏和分享。
10. 详情页SEO和结构化数据。
11. Sitemap和旧URL重定向能力。
12. 管理后台批量审核。

### 退出条件

- 新系统具备替代旧站公开分类信息的基本能力。
- 公开页面可被搜索引擎正确抓取。

## 阶段6：服务商与需求撮合

状态：第一版后置

预计时间：2至3周

### 工作步骤

1. 企业创建和成员管理。
2. 服务商主页。
3. 服务类别、地区和价格范围。
4. 资质上传和后台验证。
5. 动态服务需求问卷。
6. 需求提交和状态管理。
7. 基于分类、地区和语言的初始匹配。
8. 服务商查看需求。
9. 报价创建、查看和接受。
10. 预约创建和状态管理。
11. 真实交易关联评价。
12. 举报和审核流程。

### 初始匹配规则

```text
必要条件：
- 服务分类匹配
- 国家匹配
- 服务商状态有效
- 资质符合分类要求

评分条件：
- 地区匹配
- 语言匹配
- 响应速度
- 评价分数
- 最近活跃
- 当前接单能力
```

第一版使用可解释规则，不使用复杂机器学习。

## 阶段7：消息和通知

预计时间：1至2周

### 工作步骤

1. 基于分类信息和服务需求创建会话。
2. 文本和图片消息。
3. 未读状态。
4. 会话静音、退出和拉黑。
5. 邮件提醒。
6. 站内通知。
7. Web Push。
8. 敏感信息和垃圾消息限制。
9. 客服会话入口。

第一版可使用轮询或Server-Sent Events，不要求完整WebSocket即时通讯。

## 阶段8：支付和商业化

状态：第一版不实施

预计时间：2周

### 前置条件

- 明确收费对象和退款规则。
- 完成葡萄牙税务和消费者规则审查。
- 确认法律、保险、金融和房地产推荐费限制。

### 工作步骤

1. Stripe客户和支付会话。
2. 服务商会员套餐。
3. 线索购买。
4. 分类信息置顶。
5. 订单和支付状态。
6. 优惠码。
7. 退款。
8. 支付Webhook幂等处理。
9. 财务后台和导出。
10. 订阅到期和失败提醒。

支付卡数据不进入宝马街数据库。

## 阶段9：PWA

预计时间：1周

### 工作步骤

1. Web App Manifest。
2. 图标和启动画面。
3. Service Worker。
4. 离线基础页面。
5. 静态资源缓存策略。
6. 发布草稿本地保存。
7. 安装引导。
8. Web Push权限流程。
9. 深度链接。
10. iOS和Android主流浏览器验证。

原生App封装在业务稳定后独立排期。

## 阶段10：完整数据迁移

第一版将本阶段替换为“公开页面抓取导入”。数据库和API迁移保留为未来方案。

预计时间：1至2周

### 迁移优先级

第一版：

1. 公开页面抓取。

未来获得旧系统访问条件后：

1. 数据库直接导出。
2. 旧系统只读导出API。

### 工作步骤

1. 冻结字段映射和清洗规则。
2. 编写可重复执行的导入程序。
3. 执行完整测试导入。
4. 核对总数、抽样内容和关联关系。
5. 修复重复、缺图和无效地区。
6. 不迁移用户和密码Hash；导入内容关联系统导入主体。
7. 迁移图片并验证校验值。
8. 迁移分类信息、状态和有效期。
9. 建立旧URL重定向表。
10. 执行性能测试。
11. 上线前全量导入。
12. 增量同步。
13. 旧站短期只读。
14. 最终增量导入。

### 数据核对

- 记录总数
- 各状态记录数
- 各分类记录数
- 有图片记录比例
- 用户与内容关联率
- 重复手机号和邮箱
- 无效地区和分类数量
- URL重定向覆盖率

## 阶段11：测试、安全和上线准备

预计时间：1至2周，可与后期开发并行

### 功能测试

- 注册登录
- 发布和审核
- 搜索筛选
- 需求和报价
- 消息通知
- 评价举报
- 支付退款

### 非功能测试

- 权限和越权
- 速率限制
- 文件上传
- XSS、CSRF和SQL注入
- 数据导出和删除
- API负载
- 搜索性能
- 图片性能
- 备份恢复

### 上线清单

1. 生产环境配置核对。
2. 数据库备份。
3. 域名和TLS。
4. Cloudflare缓存和WAF。
5. robots.txt和Sitemap。
6. 邮件、短信和推送。
7. Stripe生产配置。
8. 隐私政策、用户协议和Cookie政策。
9. 客服和举报渠道。
10. 监控告警。
11. 回滚方案。
12. 运营值班安排。

## 阶段12：灰度上线和稳定期

预计时间：2至4周

### 灰度步骤

1. 内部员工和少量服务商测试。
2. 邀请新测试用户验证注册和消息。
3. 开放只读公开页面。
4. 开放发布和需求提交。
5. 开放报价和消息。
6. 支付不在第一版开放。

### 每日监控指标

- 5xx错误率
- API P95
- 登录成功率
- 发布完成率
- 搜索无结果率
- 图片上传失败率
- 消息发送失败率
- 支付失败率
- 举报数量
- 404和重定向错误

## 3. 建议里程碑

| 里程碑 | 结果 |
|---|---|
| M0 | 产品范围和业务规则确认 |
| M1 | 页面审计和试抓取完成 |
| M2 | 原型和设计系统通过评审 |
| M3 | 基础工程、账号和媒体可用 |
| M4 | 资讯和分类信息可替代旧站 |
| M5 | 基础站内消息完成 |
| M6 | 新首页、列表、详情和PWA基础完成 |
| M7 | 全量公开数据抓取和预发布测试完成 |

## 4. 团队建议

最小完整团队：

- 产品/项目负责人：1人
- UI/UX设计：1人
- 前端工程师：1至2人
- 后端工程师：1至2人
- 测试：1人，可阶段性参与
- 内容与审核运营：至少1人
- 法务和财税顾问：按需

## 5. 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 无法取得旧数据库 | 无法迁移账号和私有状态 | 第一版明确只导入公开内容 |
| 旧密码不可兼容 | 用户无法登录 | 首次登录重置密码或兼容旧Hash |
| 分类和地区数据混乱 | 搜索和迁移质量差 | 建立映射表和人工复核队列 |
| 旧图片大量缺失 | 内容质量下降 | 迁移校验、占位图和用户补图 |
| 服务商资质不真实 | 信任和法律风险 | 人工认证、有效期和审计日志 |
| 消息垃圾和诈骗 | 用户流失 | 限流、举报、拉黑和风控规则 |
| 过早开发原生App | 工期失控 | 先验证PWA和业务闭环 |
| 多国同时上线 | 合规和运营失控 | 葡萄牙验证后逐国复制 |
| 公开联系方式被滥用 | 垃圾信息和诈骗风险 | 点击展示、限流、举报、认领后可隐藏 |

## 6. 开发开始前的必备材料

1. 现有公开页面模板和URL范围。
2. 当前分类和地区清单。
3. 允许抓取、重新展示的字段确认。
4. 当前用户协议和隐私政策。
5. 公开联系方式展示及申请下架规则。
6. MVP范围的书面确认。

旧数据库、旧图片存储和旧后台访问不是第一版前置条件，但缺失意味着无法迁移旧账号、私有状态和完整历史数据。
